# Game 项目协作约定

## Project 目录概览

- 本仓库是 Go 模块 `github.com/legamerdc/game`。
- 顶层目录大致分为两类：可复用基础库，以及正在演进中的游戏引擎/并行 tick 设计。
- 当任务同时涉及代码和文档时，默认以代码表示当前行为，以文档表示当前设计意图；除非用户明确要求对齐其中一侧。

## 核心原则

1. **代码优先** - 文档与代码不一致时，以代码的实际行为为准
2. **自主分析** - 尽可能完成上下文载入，理解文档和代码，进行分析与推荐，减少不必要的交互轮次
3. **主动挑战** - 对模糊/冲突/违反架构设计的开发者要求提出质疑，帮助开发者聚焦
4. **确认先行** - 评估发现合理开发计划时，先让开发者确认大体方向，确认后才能开发
5. **及时总结** - 阶段性完成开发任务后应当及时使用subagent修改设计/记忆文档，减少文档和代码的不一致性

## 顶层目录说明

- `en/`：引擎接口与运行时抽象。当前并行 tick 讨论以 `en/world.go` 为接口权威。
- `bt/`：行为树库及其 blackboard 支持。
- `cc/`：表达式解析/编译模块，用于规则或玩法表达式求值。
- `lib/`：通用数据结构和底层工具。
- `tag/`：层级标签系统。
- `docs/`：设计稿、调研资料、讨论沉淀和协作记忆。只要讨论改变了项目理解，优先考虑是否需要更新这里。

## docs 目录约定

编写文档主体使用中文，但设计绘制文本图案时使用英文（避免markdown渲染问题）

- `docs/design/`：设计草稿、接口意图、权衡分析、设计评审记录。
- `docs/references/`：理论笔记、调研材料、外部依据，用来支撑设计，但不直接覆盖代码语义。
- `docs/memory/`：协作记忆目录，包含下方定义的三个文件。不要把长期记忆和临时草稿混在一起。
- `docs/papers/`：为发表 gdc 会议所制作的文档和草稿。

## Memory 文件结构

`docs/memory/` 目录下包含三个文件，各自职责明确，不要混合：

| 文件 | 职责 | 更新频率 |
|------|------|----------|
| `memory.md` | 稳定上下文：当前状态、已确认决策、开放问题、关键文件索引 | 决策确认或工作主线变化时 |
| `tasks.md` | 项目级任务注册表：所有被追踪的任务及其状态 | 任务状态变化时 |
| `todo.md` | 当前活跃任务的执行清单：具体步骤和进展 | 每次工作 session |

只有当这三个文件明显变得难以扫描时，才考虑新增文件（如 `decisions.md` 或 archive 目录）。

## Memory 读取规则

- **会话开始时**：必须读取 `docs/memory/memory.md`，了解当前项目状态和上下文。
- **开始执行任务前**：读取 `docs/memory/tasks.md` 和 `docs/memory/todo.md`，了解任务全貌和当前进展。
- **遇到不确定上下文时**：重新读取相关 memory 文件，不要猜测。
- **用户明确要求"继续"或"接着做"时**：视为新 session，执行完整读取流程。
- 读取顺序：`memory.md` → `tasks.md` → `todo.md`（从稳定上下文到具体执行项）。

## Memory 写入规则

以下情况必须更新对应的 memory 文件：

- 用户要求"总结"。
- 某个设计结论被确认。
- 当前工作主线发生变化。
- 之前的假设被推翻。
- 出现重要决策、范围调整、阶段切换。

当用户要求"总结"时，必须同时覆盖：

- 当前讨论进展
- 已确认的结论与假设
- 未解决的问题、风险和分歧
- 基于现状更新 `tasks.md` 和 `todo.md`

即使用户没有明确说"总结"，只要触发了上述任一条件，也应该刷新对应文件。

## `memory.md` 维护方式

- 保持精炼，优先改写过时内容，不要无限追加。
- 固定包含以下区块：
  - `Current Focus`：当前讨论/工作主线
  - `Latest State`：与当前工作直接相关的项目现状
  - `Confirmed Decisions`：可跨轮次复用的稳定结论
  - `Open Questions`：尚未落定的设计或实现问题
  - `Relevant Files`：后续需要回看的关键文件
  - `Should`：用户要求遵守的设计、开发、协作原则（正向提醒）
  - `Dont's`：用户明确要求避免的做法或习惯（反向提醒）

## `tasks.md` 维护方式

- 使用 `Active`、`Backlog`、`Blocked`、`Done` 四个区块。
- 每项任务一行，描述清晰、可识别，不需要展开具体步骤。
- 任务状态变化时更新对应区块。
- 完成项移入 `Done` 并保留日期，不直接删除。
- 当计划变化时，及时删除或改写失效任务，保证列表可被信任。

## `todo.md` 维护方式

- 顶部标明当前任务名称（对应 `tasks.md` 中的某一项）。
- 列出当前任务的具体执行步骤，使用 checkbox 跟踪进展。
- 可附加 Notes 区块记录执行中的发现或决策。
- 任务切换时，整体改写此文件。
- 如果当前没有活跃任务，保持文件为空或仅包含标题。
- 任何在总结里提到的未决事项，都应该在 `tasks.md` 中有对应条目。
